실전 프롬프트 예시

프롬프트는 크게 두 단계로 나뉜다.

  1. 설계 단계 — 도메인 분석, 아키텍처 결정 (고성능 모델용)
  2. 실행 단계 — Skills 기반 반복 구현 (경량 모델용)

각 단계별로 복사해서 바로 쓸 수 있는 템플릿을 제공한다.

1단계: 도메인 설계 프롬프트 (Opus급 모델)​

1-1. 신규 도메인 분석 요청​

[프로젝트 배경]
{프로젝트 한 줄 설명}

[요구사항]
{요구사항 목록}

[현재 기술 스택]
- Backend: FastAPI + SQLAlchemy 2.0
- DB: PostgreSQL

[요청]
위 요구사항을 도메인 단위로 분리하고,
각 도메인의 책임, 주요 엔티티, 도메인 간 의존성을 명확히 설명해줘.
의존성이 있는 경우 구현 순서도 제안해줘.

사용 시점: 프로젝트 시작 또는 대형 기능 추가 전 추천 모델: Claude Opus 4.6

1-2. Task 분리 요청​

[도메인]
{도메인명}

[도메인 설명]
{도메인 책임과 범위}

[요청]
이 도메인을 독립적으로 실행 가능한 Task 단위로 분리해줘.
각 Task에 대해 다음을 포함해서 JSON 형식으로 반환해줘:
- task_id: 식별자
- title: 작업 제목
- description: 상세 설명
- dependencies: 선행 Task 목록
- complexity: "simple" | "medium" | "complex"
- suggested_model: 권장 모델 티어

사용 시점: 도메인 설계 직후, 구현 계획 수립 시 추천 모델: Claude Opus 4.6

1-3. 아키텍처 결정 요청​

[상황]
{현재 상황 및 문제}

[선택지]
A안: {설명}
B안: {설명}

[제약 조건]
- 팀 규모: {n}명
- 예상 트래픽: {규모}
- 기존 기술 스택: {스택}

[요청]
각 안의 장단점을 트레이드오프 관점에서 분석하고,
우리 상황에 맞는 최적 안을 추천해줘.
결정 기준과 근거를 명확히 설명해줘.

사용 시점: 설계 방향 결정이 필요할 때 추천 모델: Claude Opus 4.6

2단계: 실행 프롬프트 (경량 모델 + Skills)​

2-1. CRUD API 생성 프롬프트​

[Skills: FastAPI CRUD]
- FastAPI + SQLAlchemy 2.0 (async)
- Pydantic v2 스키마
- HTTPException + ErrorResponse 패턴
- pytest + httpx 테스트 포함

[Task]
다음 엔티티에 대한 CRUD API를 생성해줘:

엔티티명: {EntityName}
필드:
- id: UUID (PK, auto)
- {field1}: {type} ({constraints})
- {field2}: {type} ({constraints})
- created_at: datetime (auto)
- updated_at: datetime (auto)

[출력 형식]
파일별로 분리하여 제공:
1. models/{entity}.py
2. schemas/{entity}.py
3. services/{entity}.py
4. routers/{entity}.py
5. tests/test_{entity}.py

사용 시점: 반복적인 엔티티 CRUD 구현 추천 모델: Gemini Flash, Kimi k2.5

2-2. 테스트 코드 자동 생성​

[Skills: pytest 테스트 생성]
- pytest + httpx 비동기 패턴
- 픽스처: conftest.py 활용
- 커버리지: happy path + error cases

[Task]
다음 서비스 코드에 대한 테스트를 생성해줘:

```python
(서비스 코드 붙여넣기)
```

[요구사항]
- 각 함수별 happy path 테스트 1개 이상
- 주요 에러 케이스 (404, 400, 권한 없음) 테스트
- 테스트 함수명은 test_(기능)_(시나리오) 패턴

사용 시점: 구현 완료 후 테스트 작성 추천 모델: Kimi k2.5, 로컬 vLLM

2-3. 리팩터링 요청​

[Skills: 코드 리팩터링]
- 기술 스택: (사용 스택)
- 목표: (가독성 향상 / 성능 개선 / 중복 제거 중 선택)

[Task]
다음 코드를 리팩터링해줘:

```python
(코드 붙여넣기)
```

[리팩터링 기준]
- 함수 단일 책임 원칙 적용
- 타입 힌트 100% 추가
- 중복 로직 추출
- 기존 인터페이스(함수명, 파라미터) 유지

[출력]
변경 전/후 코드를 함께 보여주고, 변경 이유를 간략히 설명해줘.

사용 시점: 기존 코드 정리, 기술 부채 해소 추천 모델: 로컬 vLLM (반복적인 경우)

2-4. 코드 리뷰 요청​

[Skills: 코드 리뷰]
- 보안 취약점 우선 (OWASP Top 10 기준)
- 타입 힌트 및 에러 핸들링 체크
- 성능 병목 포인트 확인

[Task]
다음 PR 변경사항을 리뷰해줘:

```diff
(git diff 내용 붙여넣기)
```

[리뷰 형식]
심각도 기준으로 분류해서 제공:
🔴 Critical (즉시 수정 필요)
🟡 Warning (수정 권장)
🟢 Suggestion (선택적 개선)

각 항목: 파일명:라인, 문제 설명, 수정 방법

사용 시점: PR 제출 전 자가 리뷰 추천 모델: Grok Code

3. 세션 전환 복원 프롬프트​

세션이 바뀌었을 때 컨텍스트를 빠르게 복원하는 프롬프트다.

[컨텍스트 복원]
첨부된 CLAUDE.md를 읽고 프로젝트 현재 상태를 파악해줘.

파악 후 다음을 간략히 요약해줘:
1. 현재 진행 중인 도메인과 Task
2. 미완성 항목
3. 주요 기술 결정사항

요약이 끝나면 "{이어할 작업 설명}"을 이어서 진행해줘.

사용 시점: 새 세션 시작 직후, 오랜만에 프로젝트 재개 시 추천 모델: Claude Opus 4.6 또는 Auto

4. 프롬프트 작성 팁​

좋은 프롬프트의 3요소
  1. 컨텍스트 — 모델이 알아야 할 배경 (기술 스택, 제약 조건)
  2. 명확한 작업 — "무엇을" 해달라는 지시
  3. 출력 형식 — "어떻게" 주면 좋은지 (파일 분리, JSON, 마크다운 등)
피해야 할 패턴
  • ❌ "이 코드 고쳐줘" (너무 모호)
  • ❌ 코드 없이 코드 리뷰 요청
  • ❌ 기술 스택 명시 없이 구현 요청
  • ✅ Skills 파일로 컨텍스트를 미리 정의하고, 프롬프트는 Task에만 집중
