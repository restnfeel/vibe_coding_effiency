모델 선정 전략

모델 선정은 작업의 성격과 비용 대비 품질이라는 두 축으로 결정한다. 가장 비싼 모델이 항상 최선이 아니다. Skills가 잘 갖춰져 있다면 저렴한 모델도 충분하다.

1. 판단 기준: 두 가지 질문​

Q1. 이 작업에 독립적·창의적 판단이 필요한가?
     YES → 고성능 모델
     NO  → 경량 모델 (+ Skills)

Q2. 이 작업이 반복적으로 발생하는가?
     YES → Skills화 → 경량 모델로 오프로드
     NO  → Auto 모드로 자동 선정

2. 추천 모델 조합 (2025–2026 기준)​

OpenRouter 활용

모든 클라우드 모델은 OpenRouter를 통해 단일 API 키로 통합 관리할 수 있다. 가격 비교 후 실시간으로 모델을 교체할 수 있어 비용 최적화에 유리하다. → 03. Cursor 설정 참고

3. 비용 계층 구조​

Tier 1 — 설계·판단 (월 $20~50)
  └── Claude Opus 4.6, GPT-4o
      사용 빈도: 낮음 / 세션당 비용: 높음

Tier 2 — 코드 생성 (월 $5~15)
  └── Gemini Flash, Kimi k2.5, Grok Code
      사용 빈도: 중간 / 세션당 비용: 낮음

Tier 3 — 반복 실행 (월 $0)
  └── 로컬 vLLM (Qwen2.5-Coder 등)
      사용 빈도: 높음 / 세션당 비용: 없음

4. 모델 자동 선정 흐름​

Cursor의 Auto 모드와 Skills를 조합하면 모델 선택을 자동화할 수 있다.

도메인 Task 진입
     │
     ├── 설계·판단 필요?
     │     YES → Opus 4.6 수동 지정
     │     NO  ↓
     ├── Skills 파일 있음?
     │     YES → Auto 모드 + Skills 자동 로드 → 경량 모델
     │     NO  → Auto 모드 (Cursor 자동 선정)
     │
     └── 반복 실행 (배치)?
           YES → Opencode + Kimi / vLLM

5. Subagent 패턴으로 병렬 처리​

단일 에이전트보다 역할 분리된 Subagent 구조가 시간과 비용 모두 절약된다.

메인 에이전트 (Opus 4.6) — Task 분리 + 결과 취합
├── Subagent A (Kimi k2.5)    →  API 엔드포인트 생성
├── Subagent B (Gemini Flash)  →  테스트 코드 생성
└── Subagent C (로컬 vLLM)    →  문서 자동화

장점:

  • 메인 에이전트 컨텍스트가 절약된다 (비용 감소)
  • 여러 작업이 병렬로 실행된다 (속도 향상)
  • 각 Subagent에 특화 Skills를 적용할 수 있다 (품질 유지)

6. 모델 선정 실수 패턴​

흔한 실수

실수 1: 모든 작업에 Opus 사용 → 비용이 5~10배 증가하고, 단순 작업은 오히려 더 느려질 수 있음

실수 2: 경량 모델에 Skills 없이 복잡한 작업 요청 → 불완전한 출력이 반복되어 재작업 비용 발생

실수 3: 세션을 너무 길게 유지 → 컨텍스트가 길어질수록 비용이 급증. 도메인 단위로 세션을 나눠라.
